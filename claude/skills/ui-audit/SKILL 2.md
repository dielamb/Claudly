# UI Audit Skill

Runs the 5 design audit frameworks against screenshot images and saves results to a structured report.

## Available audits

| ID | Name | Prompt file |
|----|------|-------------|
| `hierarchy` | Visual Hierarchy Surgeon | `prompts/1-hierarchy.md` |
| `typography` | Typography Director | `prompts/2-typography.md` |
| `spacing` | Whitespace Pressure Test | `prompts/3-spacing.md` |
| `color` | Color & Contrast | `prompts/4-color.md` |
| `diagnosis` | Brutal Diagnosis | `prompts/5-diagnosis.md` |
| `all` | Run all 5 in parallel | (default when args don't name a specific audit) |

## Usage patterns

The user invokes this skill several ways:

1. **`/ui-audit image1.png image2.png image3.png`** — run all 5 audits on those images, spawn 5 agents in parallel
2. **`/ui-audit all image1.png`** — explicit "all 5"
3. **`/ui-audit hierarchy image1.png image2.png`** — run ONE audit on those images
4. **`/ui-audit typography image1.png`** — run ONE audit
5. **`/ui-audit --list`** — list available audits and exit, no run
6. **`/ui-audit`** (no args) — interactive: ask which audit(s) and which images

Audit IDs accept short aliases: `h`/`hierarchy`, `t`/`typography`, `s`/`spacing`, `c`/`color`, `d`/`diagnosis`, `all`.

## Execution steps

### Step 1 — Parse arguments

Inspect the arguments passed to the skill. Determine:
- **Audit selection:** `all` (default) or one of `hierarchy`/`typography`/`spacing`/`color`/`diagnosis`
- **Image paths:** any arguments that look like file paths ending in `.png`, `.jpg`, `.jpeg`, `.webp` or that resolve via shell glob

If the user passed `--list`, output the audit table from the section above and exit.

If no image paths are provided, ask the user which images to audit (interactive mode). Do not fabricate paths.

If image paths were provided but don't exist on disk, report them and exit without running audits. Verify with `ls` or `test -f`.

### Step 2 — Load prompts

For each selected audit, read the corresponding prompt file from `~/.claude/skills/ui-audit/prompts/`:

- `hierarchy` → `1-hierarchy.md`
- `typography` → `2-typography.md`
- `spacing` → `3-spacing.md`
- `color` → `4-color.md`
- `diagnosis` → `5-diagnosis.md`

Use the `Read` tool to load each prompt file's content.

### Step 3 — Spawn audit agents

For each selected audit, spawn a `general-purpose` Agent in a single parallel batch (all agents in one message with multiple Agent tool calls if running multiple audits).

Each agent receives:
- The full prompt content from the prompt file
- Explicit instructions to Read each image file at the paths provided by the user
- Formatting instructions (keep output structured per the format block in the prompt, under 1200 words, concrete values only)

Example agent prompt skeleton:

```
{prompt file content}

<images_to_analyze>
Read these files using the Read tool:
- {image_path_1}
- {image_path_2}
- ...
</images_to_analyze>

<return_requirement>
Return the full markdown-formatted audit result exactly as specified in the prompt's "Output format" section. Do NOT include commentary before or after the audit. Start with `## Audit N —` and end with the cross-image pattern section.
</return_requirement>
```

When running `all`, spawn 5 agents in parallel in one message (5 Agent tool uses inside one assistant response). Wait for all to complete.

When running a single audit, spawn 1 agent.

### Step 4 — Assemble report

Once all agents return, assemble a single markdown report:

```markdown
# UI Audit Report

**Date:** {ISO date + time}
**Project:** {derived from cwd basename, or "standalone" if run from home}
**Images audited:**
- {image_1_path} ({size if available})
- {image_2_path}
- ...
**Audits run:** {comma-separated list of audit IDs}

---

{agent output 1 verbatim}

---

{agent output 2 verbatim}

---

... etc ...

---

## Consolidated findings

{If more than one audit was run, synthesize a short cross-audit summary:
- Which problems appear in multiple audits (convergent findings — high confidence)
- Which single change would address the most cross-audit findings
- Single highest-impact fix across all audits run}

{If only one audit ran, skip this section.}

---

*Generated by `/ui-audit` skill*
```

### Step 5 — Save outputs

Save the assembled report to **two locations** (or one if project has no `.planning/`):

**Primary (always):** Obsidian vault
```
~/Desktop/Labirynt/3 Atlas/Design/audits/YYYY-MM-DD-HHMM-ui-audit-{project-slug}.md
```

Where:
- `YYYY-MM-DD-HHMM` is current timestamp
- `{project-slug}` is basename of cwd, kebab-cased (e.g., `www_v2` → `www-v2`), or `standalone` if cwd is home

**Secondary (if `.planning/` exists in cwd or any parent up to 3 levels):**
```
{project_root}/.planning/audits/YYYY-MM-DD-HHMM-ui-audit.md
```

Use `Write` tool for both. Create parent directories if needed via `Bash mkdir -p`.

**Obsidian frontmatter** (primary file):
```yaml
---
type: ui-audit
created: YYYY-MM-DD
tags:
  - ui-audit
  - design-review
  - {project-slug}
project: "[[{project-slug}]]"
audits_run:
  - {audit_id_1}
  - {audit_id_2}
images:
  - {image_1_path}
  - {image_2_path}
---
```

### Step 6 — Return summary

After writing files, return a concise inline response to the user:

```markdown
# UI Audit Complete

**Audits run:** {list}
**Images:** {count}
**Report saved to:**
- `{obsidian_path}` (primary)
- `{project_path}` (secondary) ← only if applicable

## TL;DR across all audits

{2-3 sentences summarizing the single highest-impact finding from the consolidated section. No bullet points, no re-listing every audit.}

## Read the full report

To see all {N} audit outputs with concrete fixes, open the file above.
```

Do NOT dump the full audit content inline in the chat response — the user explicitly wants it saved separately. Inline response is summary + file paths only.

## Argument parsing rules

- First arg that matches an audit ID (`hierarchy`, `typography`, `spacing`, `color`, `diagnosis`, `all`, or their single-letter aliases) is the audit selection
- If first arg is a file path (has extension), treat selection as `all` and this arg as image 1
- All other args are treated as image paths
- `--list` anywhere in args → show audit table and exit
- Empty args → interactive mode (ask user)

## Error handling

- Image file not found → list missing files, exit without running audits
- Prompt file missing → report which one, instruct user to reinstall skill, exit
- Obsidian vault folder missing → create it via `mkdir -p`, continue
- Agent returns empty or malformed output → retry once with same prompt, if still fails, include the raw output under a `## FAILED AUDIT` section in the report

## Model selection

Spawn audit agents with the orchestrator's inherited model (sonnet or opus depending on current session). No explicit model override.

## Example invocations (how to route)

- `/ui-audit /tmp/hero.png` → select=all, images=[/tmp/hero.png], spawn 5 parallel
- `/ui-audit hierarchy /tmp/hero.png /tmp/hero-mobile.png` → select=hierarchy, images=[2 paths], spawn 1
- `/ui-audit all ~/Desktop/screenshots/*.png` → select=all, images=[glob-expanded], spawn 5
- `/ui-audit --list` → print audit table, exit
- `/ui-audit` → ask "which audit(s)?" then "which images?"
- `/ui-audit c /tmp/a.png /tmp/b.png` → select=color (alias c), spawn 1
